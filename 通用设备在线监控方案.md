# 通用设备在线监控方案

## 一、问题背景

当前前端通过 `$SYS` 主题监控设备在线状态，存在两个致命缺陷：

### 1. 时序依赖导致状态丢失

`$SYS` 消息仅在设备连接/断开的瞬间发布一次，**无 retained 标志**，不保留历史状态。前端连接晚于设备时，错过所有已发布的事件，已在线设备永远显示为"离线/未知"。前端断线重连后同样无法恢复——事件已消逝，不会重发。

```
设备上线 → $SYS/connected 发布并消失
  ... 时间流逝 ...
前端上线 → 订阅 $SYS → 只能收到订阅之后的未来事件
         → 之前已连接的设备状态永远为空
```

### 2. 依赖 broker 类型

`$SYS` 主题格式因 broker 而异（EMQX、Mosquitto、HiveMQ 各不同），换 broker 需要改代码，不通用。

### 用户可见表现

- 添加设备时"自动发现"列表始终为空
- 侧边栏中已在线设备始终显示为离线/未知状态

---

## 二、目标

| 角色 | 目标 |
|------|------|
| **设备端** | 按本方案设置 Will Message + 发布 retained 状态消息，覆盖正常上线、正常下线、异常断开三种场景 |
| **前端** | 通过通配符订阅设备状态主题，利用 retained 机制初始化全量状态 + 持续监听增量更新，不再依赖 `$SYS` |

**核心原则：** 设备在线状态由设备自身通过 retained 消息上报，不再依赖 broker 内部的 `$SYS` 事件。利用 MQTT 协议的 retained 消息天然持久化特性，消灭时序问题——前端任何时候订阅都能立即拿到每台设备的最新状态。

---

## 三、解决方案

### 核心思路

**将设备状态主题置于 Will 机制下，利用 Will Message 与 Retained 消息实现实时在线状态监控。**

```
设备 CONNECT（携带 Will → offline）
  │
  ├─ CONNACK 成功 → 立即发布 retained → online
  │
  ├─ 正常运行中 ...
  │
  ├─ 正常 SHUTDOWN → 发布 retained → offline → DISCONNECT
  │   （正常 DISCONNECT，Will 不会触发）
  │
  └─ 异常断开（崩溃/断电/断网）→ keepalive 超时 → Broker 自动发布 Will → offline
```

**前端侧：** 通配符订阅后，Broker 立即推送所有 retained 消息（即所有设备最新状态），后续实时增量更新——无需轮询、无需 API、无时序问题。

### 设备端需要实现的 3 个步骤

#### 步骤 1：CONNECT 时携带 Will Message

```
connect({
  will: {
    topic:     "<设备状态主题>",
    payload:   {"status":"offline","deviceId":"<deviceId>","ts":<unix毫秒>},
    qos:       1,
    retain:    true    ← 必须
  }
})
```

**作用：** 设备异常断开（崩溃/断电/网线拔了）时，Broker 代为发布这条消息，前端立即感知离线。

#### 步骤 2：连接成功后立即发布 Online 状态

```
CONNACK 成功 → 立刻 PUBLISH:

  topic:   "<设备状态主题>"      （与 Will 同一主题）
  payload: {"status":"online","deviceId":"<deviceId>","ts":<unix毫秒>}
  qos:     1
  retain:  true    ← 必须
```

**作用：** broker 保留此消息。此后任何时候任何订阅者订阅此主题（包括前端重连），都能立即收到最新的 "online" 状态。也覆盖了 Will 留下的 "offline" 残留（设备崩溃后重启的场景）。

#### 步骤 3：正常断开前发布 Offline 状态

```go
// Go 伪代码
func gracefulShutdown() {
    publish(topic, `{"status":"offline","deviceId":"...","ts":<ts>}`, qos=1, retain=true)
    disconnect()  // 正常 DISCONNECT，Will 不会触发
}
```

**作用：** 正常关机/重启时主动告知前端下线，而非依赖 Will（Will 只处理异常断开）。

### 行为矩阵

| 场景 | 前端看到 | 触发机制 |
|------|---------|---------|
| 设备上线 | online | 步骤 2 |
| 设备正常下线 | offline | 步骤 3 |
| 设备异常崩溃 | offline | 步骤 1（Will） |
| 前端重启/重连 | 全部设备最新状态 | retained 消息在订阅瞬间推送 |
| 设备崩溃后重启 | online | 步骤 2 的 retained 覆盖步骤 1 的 Will |

### 注意事项

- **retain 标志必须为 true**（步骤 1 的 Will 和步骤 2、3 的消息），否则不持久化
- **Will 和状态消息使用同一主题**，步骤 2 的 online 消息自动覆盖步骤 1 的 offline Will 残留
- **payload 至少包含** `status`（"online"/"offline"）、`deviceId`、`ts`（unix 毫秒时间戳）
- **QoS 设为 1**，至少投递一次
- **topic 命名由设备方自行定义**，双方对齐即可，前端通过通配符订阅
