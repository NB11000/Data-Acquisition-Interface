## 一、背景（Background）

当前前端可以对设备下发控制命令（打开采集卡、开始采集、连接激光器、开关激光），但无法查看或修改设备运行参数。用户只能通过后端直接操作配置文件才能调整采集卡量程、激光功率、算法系数等参数，操作路径长、效率低。

### 问题（Problem）

#### 1. 无配置可视化入口

状态栏 Row2 仅有采集卡和激光器的开关类操作按钮，缺少参数设置入口。用户无法在前端查看当前采集卡同步通道、采样频率、触发源、量程等运行时配置。

#### 2. 无统一配置界面

MQTT 文档 §2.5 定义了 9 个 RPC 配置方法（4 类硬件配置），但前端既无对应的类型定义，也无读取/写入的 UI 组件。

#### 3. RPC 响应层不支持自定义数据载荷

`CommandResult` 接口仅有 `state?: SystemStateDto` 字段承载响应体，无法适配配置读取返回的 `CaptureCardConfig` / `RadarConfig` 等非系统状态数据结构。

#### 4. Mock 模式未覆盖配置相关 RPC

`mockRpc.ts` 的 `handleMockRpc` switch 不含 9 个配置方法（`collector-config-read` 等），Mock 模式下配置读写无响应。

---

## 二、解决方案（Solution）

### 整体思路

在 StatusControlBar Row2 最右侧新增"设置参数"按钮，点击弹出 ConfigModal 模态框。Modal 通过 Ant Design Tabs 分页展示四个配置类别（采集卡/激光雷达/算法参数/持久化），打开时并发 RPC 读取当前值填充表单，各 Tab 独立保存。底层扩展 `CommandResult.data` 字段承载任意响应载荷。

### 具体实施

#### 1. RPC 类型层扩展

`CommandResult` 接口新增 `data?: unknown` 字段，使配置读写响应能携带自定义数据体。不修改 `tryResolveRpc`（JSON.parse 自然兼容多出字段），不修改 `sendRpcCommand` 签名。

```typescript
// src/mqtt/types.ts
export interface CommandResult {
  success: boolean;
  code: string;
  message: string;
  data?: unknown;  // 新增：配置读写等通用载荷
  state?: SystemStateDto;
  timestamp: string;
}
```

#### 2. Mock 补齐 9 个配置方法

`handleMockRpc` 新增参数 `payload?: unknown`，在 switch 中追加 9 个 case（读返回硬编码默认值、写回显请求体）：

| 方法 | 行为 |
|------|------|
| `collector-config-read` | 返回默认 CaptureCardConfig（7 字段） |
| `collector-config-update` | 回显 payload 到 data |
| `collector-config-default` | 返回默认值，共用 defaultCollectorConfig 常量 |
| `laser-config-read` | 返回默认 RadarConfig（4 字段） |
| `laser-config-update` | 回显 payload |
| `lidar-config-read` | 返回默认 LidarAlgorithmConfig（11 字段） |
| `lidar-config-update` | 回显 payload |
| `persistence-config-read` | 返回默认 PersistenceSettings（1 字段） |
| `persistence-config-update` | 回显 payload |

#### 3. ConfigModal 组件

**文件**：`src/pages/Dashboard/components/ConfigModal.tsx`（336 行）

- **Props**：`{ open: boolean; onClose: () => void }`
- **Modal**：标题"设备参数配置"，宽度 640px，`destroyOnHidden`
- **4 个 Tab**：采集卡 / 激光雷达 / 算法参数 / 持久化
- **配置接口**：`CaptureCardConfig`（7 字段）、`RadarConfig`（4 字段）、`LidarAlgorithmConfig`（11 字段）、`PersistenceSettings`（1 字段）——与 MQTT 文档 §2.5 严格对齐
- **下拉常量**：`SYNC_CHANNEL_OPTIONS`、`CLOCK_SOURCE_OPTIONS`、`HALF_FULL_OPTIONS`、`TRIGGER_SOURCE_OPTIONS`、`RANGE_OPTIONS`、`BAUD_RATE_OPTIONS`——全部硬编码

**读取流程**：弹窗打开 → useEffect 触发 → 4 个 Tab 并发调用 `sendCommand` → 成功填充 `Form.setFieldsValue` / 失败显示 error Alert + 重试按钮

**保存流程**：点击 Tab 底部"保存" → `Form.validateFields()` → 校验通过 → `sendCommand(updateMethods[key], values)` → 成功 `message.success` / 失败 `message.error`

**状态管理**：组件内 `useState`（loadState、saving），无需新增 Zustand store。

#### 4. StatusControlBar 按钮集成

Row2 激光器组后新增第三个操作组：

```tsx
<Button size="small" icon={<SettingOutlined />} disabled={allDisabled}
  onClick={() => setConfigModalOpen(true)}>设置参数</Button>
```

复用现有 `allDisabled` 逻辑（MQTT 断开 / will 崩溃 / 未选中设备时置灰）。Fragment 包裹原 JSX，Modal 渲染在 Fragment 末尾。

| `StatusControlBar.test.tsx` | 7 | 按钮显示、5种 disabled 条件、弹出弹窗 |

#### 6. 测试基础设施

- 新增 `@testing-library/react`、`@testing-library/jest-dom`、`@testing-library/user-event`、`jsdom` 依赖
