## 一、背景（Background）

前端当前代码采用"单设备-单连接"MQTT 架构，Device 虽然携带各自的 brokerUrl/port/username/password，但实际 MQTT 连接仅依赖 `.env` 全局变量，多 Broker 场景不可用。代码中 Mock 和 Real 模式通过散落的 `if (MQTT_MODE)` 分支混合在一起。

需向前一步推进到"多服务器-多连接"架构：前端为每个 MQTT 服务器维护独立连接，同服务器设备共享连接，跨服务器设备独立连接。该变更触及数据模型、连接管理、RPC 路由、UI 组件、Mock/Real 解耦等多个维度，需先完成完整的领域建模和实施方案设计。

### 问题（Problem）

#### 1. 设备模型与 Broker 认知脱节

现有 Device 模型嵌入完整连接配置（brokerUrl/port/username/password/tls/caCert），但代码中实际只使用 `.env` 全局配置。ADR 0001 确立了"每设备独立配置"路线，但在实践中该设计从未落地。

#### 2. 缺乏多服务器架构的完整设计方案

CONTEXT.md 描述了"计划中的连接池架构"，但没有显式 MqttServer 实体、没有完整的连接生命周期定义、没有跨服务器设备切换逻辑、没有 Mock/Real 解耦方案。

#### 3. Mock/Real 模式代码耦合

`rpc.ts`、`useMqttConnect.ts`、`client.ts` 等多处包含 `if (MQTT_MODE === 'mock')` 分支，两种模式的逻辑交织在同一个函数体内，Mock 修改可能影响 Real，真实模式下代码路径仍需经过 mock 判断。

#### 4. 缺乏可执行的实施路线

没有 PRD、没有实现计划、没有分解好的 Issue，工程团队无法按迭代推进。

## 二、解决方案（Solution）

### 整体思路

通过 grilling-with-docs 工作流完成领域建模与 33 个决策点的逐一推敲，产出更新后的 CONTEXT.md、新 ADRs、完整 spec、PRD、实现计划和 8 个垂直切片 Issue。

### 具体实施

#### 1. 领域模型重定义（CONTEXT.md 重写）

从"每 Device 自包含连接配置"转为"显式 MqttServer + 轻量 Device 引用"：

```
旧: Device { id, name, brokerUrl, port, username, password, tls, caCert, isOnline }
新: Device { id, name, serverId, isOnline }
    MqttServer { id, name, brokerUrl, port, username, password, tls, caCert, isConnected }
```

连接池 key 从 `brokerUrl:port:username:password` 隐式哈希 → `serverId` 显式引用。新增 5 种连接状态（已连接/已断开/重连中/连接失败/初始化中）、自动重连策略（3 次 / 指数退避 1s→2s→4s）、Online Client Cache 等概念。

#### 2. 连接生命周期完整建模

生命周期表从原有的 7 行扩展到 14 行，覆盖页面初始化、服务器/设备 CRUD、同服务器/跨服务器设备切换、连接失败/重连/重连期间设备切换、页面关闭等所有场景。明确了服务器删除确认框、编辑服务器名称热更新 vs 连接参数重连、空服务器断连等边界行为的交易语义。

#### 3. Mock/Real 依赖注入架构（ADR 0004）

5 层分层设计：

```
接口层:  MqttClientLike, MqttConnectionFactory
实现层:  RealMqttClient, MockMqttClient
连接池:  ConnectionPool (只依赖接口)
业务层:  RPC, Router, Stores, Hooks
入口层:  AppEntry (VITE_MQTT_MODE 选择工厂)
```

MockMqttClient 内建 topic router（`Map<topic, handler[]>`），subscribe 注册/unsubscribe 移除/publish 分派，行为与真实 MQTT 一致。全代码树无 `if (MQTT_MODE)` 分支。

#### 4. 项目文档体系建设（AGENTS.md 扩展）

追加项目技术栈（React 19 + TS 6 + Vite 8 + Ant Design 6 + ECharts 6 + MQTT + Zustand 5）、pnpm 命令、架构概述（SPA/路由/Store/MQTT 单例/Router 优先级）、上下文与代码差距说明、CSS Modules 约定、Mock 模式陷阱、环境变量说明、测试基础设施。

#### 5. 实施路线分解（PRD + Plan + 8 Issues）

- **PRD**: 25 个 User Stories + 实现决策 + 测试策略 + Out of Scope
- **实现计划**: 16 个模块、接口定义、数据流（含 Happy/Error/Edge 路径）、11 项关键技术决策、测试策略、9 个垂直切片设计
- **8 个 Issues**: 严格按 TDD 红绿重构循环约束，按依赖顺序排列（数据模型→连接池→Router/RPC→添加设备→设置页面→侧边栏→状态监控→环境清理）
