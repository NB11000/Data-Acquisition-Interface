## 一、背景（Background）

上次提交（docs(arch)）建立了"多服务器 MQTT 连接架构"的完整设计文档体系（CONTEXT.md、3 个 ADR、PRD、实现计划、8 个 Issue），但代码仍停留在"单设备-单连接"架构。本次提交将设计落地为代码实现，完成全部 8 个 TDD 垂直切片 Issue。

### 问题（Problem）

#### 1. 单例 MQTT 连接无法支持多 Broker

现有代码通过 `getMqttClient()` 全局单例访问唯一 MQTT 连接，即使 Device 携带各自的 brokerUrl/port/username/password，这些字段从未被使用。用户无法同时连接多台 MQTT 服务器。

#### 2. Device 模型耦合连接配置

Device 接口嵌入 brokerUrl/port/username/password/tls 等 MQTT 连接字段，每次添加设备都需重复填写完整连接参数。

#### 3. Mock/Real 模式代码耦合

`rpc.ts`、`useMqttConnect.ts`、`client.ts` 等多处包含 `if (MQTT_MODE === 'mock')` 分支，Mock 修改可能影响 Real 模式。

#### 4. 缺少 UI 基础设施

系统设置页面为空占位符；添加设备分为两个独立模态框（ManualAdd + AutoDiscover），无服务器选择概念；侧边栏为扁平列表不支持服务器分组。

#### 5. 在线状态监控不完整

`$SYS` 路由未绑定 serverId，无法区分不同服务器上线的设备；Online Client Cache 不存在。

#### 6. 废弃环境变量残留

`VITE_BROKER_URL` / `VITE_BROKER_USERNAME` / `VITE_BROKER_PASSWORD` / `VITE_DEFAULT_MACHINE_ID` 在新架构下无意义但未清理。

## 二、解决方案（Solution）

### 整体思路

按 8 个垂直切片逐层实现：数据模型与持久化 → 连接池核心 + 依赖注入 → Router/RPC 多连接改造 → UI 组件（添加设备 / 设置页面 / 侧边栏）→ 在线状态监控 → 环境清理。每个核心模块附 TDD 单元测试。

### 具体实施

#### 1. 数据模型重构（Issue 1）

Device 从 7 字段（嵌连接配置）简化为 4 字段（id/name/serverId/isOnline）；新增 `MqttServer` 实体承载连接配置。新建 `serverStore` 和改造 `deviceStore`，均使用 Zustand persist 中间件持久化到 localStorage（key: `mqttServers` / `devices`）。实现 `runMigration()` 旧格式自动转换逻辑附带 8 个测试。

```typescript
// 变更前后对比
Device 旧: { id, name, brokerUrl, port, username, password, tls, isOnline: boolean }
Device 新: { id, name, serverId, isOnline: boolean | null }
MqttServer 新: { id, name, brokerUrl, port, username, password, tls, caCert?, connectionState? }
```

#### 2. 连接池核心 + 依赖注入（Issue 2）

- **MqttClientLike 接口**：定义 MQTT 客户端抽象（connect/subscribe/unsubscribe/publish/end/isConnected + 事件回调）
- **MockMqttClient**：内建 `Map<topic, handler[]>` topic router，支持 `+` 单层通配符和 `#` 多层通配符匹配
- **ConnectionPool**：`Map<serverId, MqttClient>` + `Map<deviceId, serverId>` + `Map<serverId, Set<clientId>>` 在线客户端缓存；5 态状态机（initializing/connected/disconnected/reconnecting/failed）；自动重连（1s→2s→4s，3 次上限）；`subscribeDevice`/`unsubscribeDevice`/`switchFollowing` 主题管理；EventEmitter 模式 `onMessage`/`onStateChange`
- **ConnectionFactory**：`createMockFactory()` / `createRealFactory()` 工厂函数，App 入口根据 `VITE_MQTT_MODE` 注入

测试：ConnectionPool 6 个 + MockMqttClient 8 个 + 现有 8 个 Migration = **22 测试通过**

#### 3. Router + RPC 多连接改造（Issue 3）

- **Router**：`setupRouter(pool)` 为每条连接注册 onMessage（闭包捕捉 serverId）；分发顺序 `tryResolveRpc` → domain handlers（state/will/alarm/data）
- **RPC**：`sendRpcCommand(machineId, method, payload?)` 内部查 deviceStore → serverId → pool.publish；发送前检查连接状态 → 非 connected 即 reject `CONNECTION_LOST`；保留 10s 超时
- 标记旧版 `getMqttClient()` 为 `@deprecated`

测试：Router 7 个 + RPC 4 个 = **33 测试全部通过**

#### 4. UI 组件实现（Issues 4-7）

- **AddDeviceModal**：合并服务器选择下拉框 + 自动发现 Tab（读 OnlineClientCache）+ 手动添加 Tab；零服务器自动引导到 MqttServerModal
- **MqttServerModal**：添加/编辑服务器表单（名称/地址/端口/用户名/密码/TLS/CA 证书上传 + 测试连接）
- **Settings Page**：服务器卡片列表（5 态图标+错误摘要）+ 设备 Table；实时订阅 pool.onStateChange
- **Sidebar**：Ant Design Tree 按服务器分组 + 搜索自动展开 + 在线状态图标
- **useMqttConnect**：完全重写适配连接池，同服务器/跨服务器设备切换各走不同路径
- **Dashboard**：连接断开半透明 banner overlay

删除旧组件：`ManualAddModal.tsx`、`AutoDiscoverModal.tsx`

#### 5. 图表无限延申修复

Dashboard wrapper div 增加 `display: flex; flexDirection: column; overflow: hidden;`，使 ChartGrid 的 `flex: 1` 生效，配合 `overflow: hidden` 裁剪溢出。ECharts 实例在无约束高度容器内按内容撑开导致无限延申的问题修复。

#### 6. 环境变量清理（Issue 8）

删除 `.env` 中 VITE_BROKER_URL/VITE_BROKER_USERNAME/VITE_BROKER_PASSWORD/VITE_DEFAULT_MACHINE_ID；删除 `src/env.ts` 和 `src/vite-env.d.ts` 中对应导出和类型声明。`src/` 目录零残留引用。
