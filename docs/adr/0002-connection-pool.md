# ADR 0002: MQTT 连接池架构

**日期**: 2026-05-29
**状态**: Accepted

## 背景

前端直接与设备所属的 MQTT Broker 通信（纯前端方案，无中间层后端）。面临三个约束：

1. 设备可能连接不同 Broker（少数场景），每个 Broker 的认证凭据和 TLS 证书可能不同
2. 设备在线状态需实时监控（侧边栏显示所有设备在线/离线）
3. ACL 限制不同账号只能订阅授权 topic

## 决策

**采用连接池（Connection Pool）架构**：前端维护 `Map<connectionKey, MqttClient>`，每个设备映射到一条 MQTT 连接。连接常驻、设备切换仅做主题切换。

## 理由

**为什么不用单连接方案**（ADR 0001 的原方案）：
- 不同 Broker 的设备需要不同连接，单连接无法覆盖
- 不同账号的 ACL 隔离（A 设备账号无法订阅 B 设备的 topic）——单连接下无法解决

**为什么不用"切换设备即断连重连"方案**：
- 重连 TLS 握手有延迟（~1-2s），切换设备体验差
- 无法实时监控非选中设备的在线状态（侧边栏需要）
- Will 消息和 Alarm（Retain=true）无法感知非选中设备

**连接池的优点**：
- 常驻连接通过 `$SYS` 主题实时获取 Broker 下所有设备上下线事件，无需 HTTP API 轮询
- 选中设备切换仅做主题订阅切换（微秒级）
- 大部分场景下设备在同一 Broker 上，连接池大小 = 1，退化到极简形态

## 分组 key

`brokerUrl + port + username + password`。`tls` 由 port 隐含（1883→非TLS, 8883→TLS），`caCert` 不进 key（第一个连上的设备提供 CA，同 Broker 用同一 CA）。

## 代价

- 需要维护多连接生命周期（建立/销毁/失败重试）
- 连接失败时对应设备 isOnline = null（未知），需 UI 区分

## 替代方案

1. **HTTP API 轮询在线状态** — 需要用户额外配置 EMQX API Key，增加添加流程复杂度，且 5s 轮询有延迟。被否定。
2. **引入中间层后端** — 属于另一架构方向，当前阶段不接受。
