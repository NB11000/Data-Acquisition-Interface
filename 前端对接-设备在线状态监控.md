# 设备在线状态监控 — 前端对接文档

> 版本：1.0 | 更新：2026-06-03 | 基于 ADR-0004

---

## 1. 概述

设备在线状态通过 MQTT retained 消息实现，不再依赖 broker 内部的 `$SYS` 主题。前端通过通配符订阅主题，利用 retained 机制**立即**获取所有设备最新状态，后续增量更新无需轮询。

**核心机制**：同一 topic 承载三种来源的消息，retained 覆盖保证任何时候订阅方拿到的都是最新状态。

| 来源 | 触发时机 | 含义 |
|------|---------|------|
| 设备主动 publish | CONNACK 成功后 | 上线 |
| 设备主动 publish | 正常 DISCONNECT 前 | 下线 |
| Broker 代发 Will | keepalive 超时 | 崩溃下线 |

---

## 2. 主题

```
daq/{MachineId}/events/will
```

| 操作 | 主题 |
|------|------|
| 订阅所有设备 | `daq/+/events/will` |
| 订阅单台设备 | `daq/{MachineId}/events/will` |

**QoS**: 1（至少一次）  
**Retain**: true（新订阅者立即获取最新状态）

> `MachineId` 从 topic 路径**第二段**提取。payload 不携带 `deviceId`。

**提取示例 (JavaScript)**：

```typescript
const match = topic.match(/^daq\/(.+)\/events\/will$/);
const machineId = match ? match[1] : null;
```

## 3. Payload 格式

统一 6 字段 JSON，三种场景共用同一结构：

```typescript
interface DeviceStatusPayload {
  status: "online" | "offline";           // 在线状态
  ts: number;                             // Unix 毫秒时间戳
  eventType: "device_online" | "device_offline" | "process_crashed";
  source: "device" | "mqtt_broker";       // 发布来源
  message: string;                        // 可读描述
  timestamp: string;                      // UTC 时间 (ISO 8601)
}
```

### 三种场景对照

| 字段 | 主动上线 | 主动下线 | 崩溃 (Will) |
|------|---------|---------|-------------|
| `status` | `"online"` | `"offline"` | `"offline"` |
| `ts` | 当前 ms | 当前 ms | **`0`** |
| `eventType` | `"device_online"` | `"device_offline"` | `"process_crashed"` |
| `source` | `"device"` | `"device"` | `"mqtt_broker"` |
| `message` | 设备已上线 | 设备正常下线 | 设备已离线 |
| `timestamp` | 当前 UTC | 当前 UTC | **`0001-01-01T00:00:00Z`** |

### 示例 JSON

**上线**：
```json
{
  "status": "online",
  "ts": 1717400000000,
  "eventType": "device_online",
  "source": "device",
  "message": "设备已上线",
  "timestamp": "2026-06-03T10:30:00.0000000Z"
}
```

**正常下线**：
```json
{
  "status": "offline",
  "ts": 1717400300000,
  "eventType": "device_offline",
  "source": "device",
  "message": "设备正常下线",
  "timestamp": "2026-06-03T10:35:00.0000000Z"
}
```

**崩溃 (Will)**：
```json
{
  "status": "offline",
  "ts": 0,
  "eventType": "process_crashed",
  "source": "mqtt_broker",
  "message": "设备已离线",
  "timestamp": "0001-01-01T00:00:00Z"
}
```

## 4. 订阅与解析

### 连接时订阅

```typescript
mqttClient.on("connect", () => {
  mqttClient.subscribe("daq/+/events/will", { qos: 1 });
});
```

Broker 在订阅成功瞬间**立即推送所有符合通配符的 retained 消息**——每条来自不同 `{MachineId}` 的设备一条，只需遍历初始推送即可建立全量状态表。

### 消息路由

```typescript
mqttClient.on("message", (topic, payload) => {
  // 只处理 events/will 主题
  if (!topic.endsWith("/events/will")) return;

  const machineId = extractMachineId(topic);
  const data: DeviceStatusPayload = JSON.parse(payload.toString());

  updateDeviceStatus(machineId, data);
});

function extractMachineId(topic: string): string {
  const match = topic.match(/^daq\/(.+)\/events\/will$/);
  return match ? match[1] : "";
}
```

## 5. 状态映射与 UI 行为

### status 字段（主要判断依据）

| `status` | UI 状态 | 颜色 | 说明 |
|-----------|---------|------|------|
| `"online"` | 在线 | 绿色 | 设备正常连接中 |
| `"offline"` | 离线 | 灰色/黄色 | 设备不在线 |

### eventType 字段（辅助区分）

| `eventType` | UI 文案 | 图标/样式 |
|-------------|---------|----------|
| `"device_online"` | 设备已上线 | 绿色圆点，正常态 |
| `"device_offline"` | 设备已下线 | 灰色圆点，主动离开 |
| `"process_crashed"` | 设备已崩溃 | **红色圆点**，告警态 |

**判断逻辑**：

```typescript
function applyDeviceStatus(machineId: string, data: DeviceStatusPayload) {
  const status = data.status;

  if (status === "online") {
    setDeviceState(machineId, "connected");       // 绿色
  } else if (data.eventType === "process_crashed") {
    setDeviceState(machineId, "crashed");          // 红色 + 告警提示
  } else {
    setDeviceState(machineId, "disconnected");     // 灰色
  }
}
```

### 时间戳处理

- `status === "online"` 且 `eventType === "device_online"`：使用 `ts` 或 `timestamp` 显示"最后上线时间"
- `status === "offline"` 且 `eventType === "device_offline"`：使用 `ts` 或 `timestamp` 显示"最后下线时间"
- `eventType === "process_crashed"`：**忽略 `ts` 和 `timestamp`**（均为占位值/零值），使用前端本地收到消息的时间作为"崩溃探测时间"

## 6. 异常场景处理

### 前端重连

MQTT 客户端断线重连后，重新订阅 `daq/+/events/will`，Broker 再次推送所有 retained 消息 → **全量状态自动恢复**，无需额外逻辑。

### Will 与崩溃感知

| 场景 | 表现 | 窗口期 |
|------|------|--------|
| 设备主动下线 | 立即收到 `eventType:"device_offline"` | 无延迟 |
| 设备崩溃 | Broker 检测 keepalive 超时后推送 Will | **最多 ~45s** |
| publish 失败兜底 | 与崩溃路径相同（走 Will） | 最多 ~45s |

> 崩溃场景存在 ~45s 窗口期：期间主控进程已死，但 Broker 尚未检测到 keepalive 超时，前端可能短暂看到设备在线。建议对长时间无数据更新的设备显示"疑似离线"。

### 幂等性

QoS 1 可能重复投递，但同一 topic retained 覆盖机制保证最终状态正确。前端应基于 `ts`（非 Will 场景）做幂等判断——`ts` 不大于已记录值则跳过更新。

## 7. 与 state_changed 主题的关系

| 主题 | 职责 | 是否使用 |
|------|------|---------|
| `daq/{MachineId}/events/will` | **设备在线/离线/崩溃** | ✅ 使用 |
| `daq/{MachineId}/events/state_changed` | 采集卡/激光器等业务状态 | 可选（后续版本） |

`state_changed` 主题不再推送 `mqtt_connected` / `mqtt_disconnected` 事件——MQTT 连接状态的唯一推送通道是 `events/will`。

## 8. 快速参考

```
订阅主题:    daq/+/events/will
QoS:         1
Retain:      true
MachineId:   topic 第二段 "daq/{MachineId}/events/will"
状态判断:    优先看 status 字段，eventType 区分离开原因
Will 字段:   ts=0, timestamp=0001-01-01, 前端忽略改用本地时间
窗口期:      ~45s (崩溃检测延迟)
```
