数据架构方案对比

核心挑战：100ms 波形推送 (16KB/帧) + 7s 低频数据 + 状态事件 如何在 React 中高效处理？

B

分层数据架构 (推荐)

数据层用纯 TS 模块，脱离 React 生命周期。热数据 (100ms 波形) 通过 uPlot 命令式更新，温数据 (7s 低频) 通过 observable store 订阅到 leaf 组件，冷数据 (状态/配置) 走 React state。

优点

  • 热数据不进 React，零 GC 压力
  • 数据层可独立测试
  • 符合 frontend-perf 规范

缺点

  • 文件较多，初始搭建成本高
  • 团队需理解分层理念
A

纯 React State 方案

所有状态通过 useState/useReducer/Context 管理。MQTT 消息直接 setState。uPlot 通过 useRef + useEffect 更新。

优点

  • 标准 React，学习成本低
  • 代码集中，文件少

缺点

  • 100ms setState 造成高频重渲染
  • Context 变化导致大面积子树更新
  • GC 压力大 (每帧 16KB 新对象)
C

Zustand 集中式 Store

使用 Zustand 管理所有状态。MQTT 消息 dispatch action 更新 store。组件通过 selector 订阅需要的 slice。

优点

  • DevTools 可调试
  • selector 防止不必要渲染
  • 社区成熟

缺点

  • 增加依赖
  • 热数据仍然经过 store (字符串比较/序列化)
  • 对于本场景偏重
